# PPT 提示词重设计 (按 PDF 主语言输出提示词，默认逐页生图并合成 PDF)

本 skill 的目的：为用户提供的 PDF 版 PPT 输出一份**完整的中文视觉优化方案**，包括：

1. 一份整套 PPT 的视觉优化建议文档（中文，针对原稿落地的可执行建议）。
2. 每一页的生图提示词（中文 PDF 输出中文，英文 PDF 输出英文，中英混排按占比更多的语言输出）。
3. 在 Codex 中默认使用内置 `image_gen` 逐页生成图片，并把所有生成图片按页码顺序合成为一个 PDF。
4. 在 Codex 以外的平台，或用户明确要求只输出提示词/不输出 PDF 时，只交付按**每 8 页一组**合并后的「批量生图大提示词」。

**文字处理原则**：页面上所有有信息价值的文字（标题、副标题、正文、列表项、数字、图注等）必须在提示词中逐一列出具体内容，不得只写数量和层级。原稿页码仅用于内部排序，不写入生图提示词，也不出现在最终画面中。

完整的文字内容能让生图模型准确还原信息，同时版式字段负责引导排版创意。

写提示词时必须优先保证原文完整度；如果某处文字因截图模糊、裁切、乱码而无法可靠识别，必须在当前页提示词前先停止处理并说明需要更清晰来源，不能用"若干文字""几段正文""多个小标题"替代。

## 版本规则

- 当前版本号记录在根目录 `VERSION`。
- 默认每次 push 发布一个小版本：`v1.0 -> v1.1 -> v1.2`。
- 只有用户明确要求发大版本时，才升级主版本：例如 `v1.1 -> v2.0`。
- 每次运行主要脚本时，先输出当前运行版本号，方便确认实际执行的是哪一版。

## 硬性规则 (Hard Rules)

- Codex 默认完整模式必须使用内置 `image_gen` 逐页生图并合成 PDF；用户明确要求只输出提示词或不输出 PDF 时，停止在合批提示词交付。
- 在 Codex 以外的平台使用本 skill 时，固定为提示词模式：不调用生图能力，不生成 `generated_images/`，不合成 `redesigned_deck.pdf`，只交付 `batched_prompts/batch-XX.md`。
- 默认完整模式中，每个 `page_prompts/page-###.md` 的最终提示词单独调用一次生图能力，禁止把多页提示词合并为一次生图请求。
- 默认完整模式的最终交付 PDF 中，每一页图片必须来自上述生图能力调用结果；禁止通过生成 PPTX、HTML、SVG、Canvas、网页截图、本地绘图、代码渲染等非生图模型方式制作页面图，再转换成 PDF。
- 若用户明确要求只输出提示词或不输出 PDF，只生成并交付 `batched_prompts/batch-XX.md` 提示词集合，不调用生图能力，不生成 `generated_images/`，不合成 `redesigned_deck.pdf`。
- 若在 Codex 中执行默认完整模式但无法调用内置 `image_gen`，必须终止生成并向用户说明原因；不得用其他生成或渲染方式替代。
- 默认完整模式中，每张生成图片按页码保存到 `generated_images/page-###.png`；必须确认页码连续、数量与原 PDF 页数一致，再合成为 `redesigned_deck.pdf`。
- 拆出的 `source_pages/` 仅供 Agent 分析与填充提示词使用，不会被发送给任何生图工具。
- 最终的视觉优化建议文档必须用中文输出。
- 每一页的生图提示词必须按 PDF 主语言输出：中文占多则中文，英文占多则英文；中英混排时做简单占比判断即可，不追求精确。
- **必须通过视觉识别读图**：PDF 文字层提取经常不完整、漏抄或顺序错乱，Agent 必须直接打开 `source_pages/page-###.png` 像看图一样视觉识别整页内容，把页面上的全部文字与全部图片都抄入对应字段——禁止仅依赖 PDF 文字层。
- **提示词语言判断**：Agent 浏览 `source_pages/` 时顺手按可见文字判断目标提示词语言；中文占多则中文，英文占多则 English，中英混排时按占比更多的一方即可。最终所有单页提示词必须统一使用该语言，但不要把“目标提示词语言：中文/English”写入最终生图提示词。原稿中的文字内容保持原文逐一列出，不翻译页面上的原始文字。
- **图片识别（仅供内部参考，不输出到提示词）**：Agent 须视觉识别页面中的真实图片/截图/照片/插画，并将其位置与内容纳入"版式"字段的布局描述中（如"右侧配合一张产品截图"）——但**不在最终提示词中单独输出「保留图片」段落**。
- **保留范围限定为文字**：原 PPT 中的全部文字必须逐一出现在"文字概览"字段中（不得只写数量/层级，必须写出实际文案内容）。装饰性图标（icon）、几何形状、线条、卡片底纹、渐变背景、分隔条等视觉元素由 Agent 自行判断是否在"版式"字段中描述。
- **每页版式必须是简洁图形化幻灯片布局**：版式字段必须用适合横版幻灯片的信息结构来传达内容关系，不接受海报式大画面、封面式口号构图或纯文字罗列。优先选择清晰、简洁、扁平的图形化布局，例如时间轴、金字塔、流程图、循环图、漏斗、同心圆、树状图、维恩图、鱼骨图、流程箭头、对比分栏、大数据 + 注解、占比环形图等；根据内容语义自由选择最合适的结构，不限于此列表。整体保持**简约扁平**视觉特征——不要立体、不要 3D 浮块、不要拟物、不要厚重光影。
- **风格只给大方向，不写纹理与细节装饰**：最终生图提示词中的风格描述只给大方向（整体气质、字体情绪、结构母题、少量颜色名称与色彩角色），不写纹理、材质、装饰细节、具体图标、线条类型、背景图案等。中文输出，不写十六进制色值。
- **版式只给大方向，不写细节装饰**：最终生图提示词中的版式描述只给大方向（一个核心视觉结构、主要元素在画面的大致位置、色彩分区），不写节点细节、图标类型、装饰线、渐变、阴影、纹理等元素级装饰。中文输出，可呼应风格中的主色、背景色和强调色。
- **参考风格写法**：可参考「A minimalist, editorial presentation style featuring high-contrast typography (brutalist sans-serif mixed with delicate script), stark black text on an off-white background, and distinctive structural motifs like underscore prefixes and oversized slash-numbering.」的思路，用中文把它重写为「整体气质 + 字体情绪 + 结构母题 + 少量颜色名称与色彩角色」，不写材质与装饰细节。
- **版式必须从内容语义推导，而非套用固定模板**：Agent 在写"版式"字段之前，必须先分析本页的**内容语义**——这页核心信息是什么？信息单元之间是并列、流程、层级、对比、构成还是单一强调关系？——然后从这个内容结构出发，推导出最能直接传达该结构的视觉排版方案。"版式"字段的描述必须体现出「因为内容是 X 关系，所以选择 Y 排版」的因果逻辑，让生图模型感知到排版是内容驱动的，而非随机套用的。
- **排版必须与原图明显不同**：最终生图提示词所描述的排版方案，必须与原 PPT 页面的视觉布局有明显差异——原图中文字的位置、图片的位置、元素的排列方式均**不需要沿用**。Agent 应把原稿中的文字和图片视为「原材料」，而非「参考排版」，完全从内容语义出发重新设计画面结构。如发现写完的版式描述与原图雷同（如同样是左文右图、同样的标题居左、同样的卡片顺序），必须主动调整为更具创意的全新排版方案。
- **提示词只写少量颜色名称，不写排版细节参数**：最终生图提示词中，配色只写 2-4 个核心颜色及其角色（如主色、背景色、强调色），不写十六进制色值；字号大小、字号倍率、百分比占比、具体像素/尺寸、Bold / Regular 等字重描述一律不写，留给生图模型自由发挥。
- **禁止卡片式排版**：最终生图提示词中，不得出现任何卡片式设计的版式描述——包括卡片矩阵、信息卡片、圆角卡片、卡片列表、网格卡片等。对于并列关系的内容，改用无边框的纯色色块分区、分栏对照、图标 + 文字组合、横向并排色带等非卡片方式呈现。
- **全套风格必须一致**：Agent 在 Step 3 完成 `deck_style_brief.md` 后，必须提炼出一段统一的「风格语句」，并将**完全相同**的风格语句复制到每一页提示词的「风格」字段——不得按单页内容随意改变整体视觉调性（如某页改成"科技发布会风"、另一页改成"温暖人文风"）。风格字段的整体定位与气质描述必须全套统一，仅配色的强调色可根据页面内容做轻微调整。
- **提示词应面向横幅幻灯片输出**：最终生图提示词允许并鼓励使用「横幅幻灯片」描述图片类型和画面目标，但不得出现「原 PPT」「原稿」「第 X 页」等来源痕迹，也不得要求照搬原始页面布局。
- **每页提示词必须以图片类型词开头**：每个最终生图提示词正文的第一行必须只写「横幅幻灯片」五个字，用来声明图片类型；第二行开始再写「风格」「版式」「文字概览」。不要写成「生成一张横幅幻灯片」或其他动作句。
- 不允许在提示词中虚构原稿没有的文字（不发明新标题、副标题、口号、按钮文案、图例等）。
- 必须把所有页的提示词**按每 8 页一组**打包成大提示词文件 `batched_prompts/batch-01.md`、`batch-02.md`……。最后一组不足 8 页时按实际页数生成。
- 单页提示词必须自包含：第一行写「横幅幻灯片」，并包含风格、版式、文字概览、输出比例。最终提示词中不出现「禁止 xxx」文案，不写“目标提示词语言”，也不写“生成一张横幅幻灯片”这类生成动作句。

## 工作流 (Workflow)

### Step 0. 更新检查

每次使用本 skill 时，先运行：

```bash
python3 scripts/update_skill.py
```

安装模式由脚本自动判断：

- 下载版安装（默认）：跳过远端更新检查，直接使用当前本地版本继续。
- Git clone 安装：每次使用前从远端检查并快进更新；如果未先完成更新检查，后续 PDF 处理脚本会停止。

如果 Git 版脚本提示已更新，必须重新读取当前目录下最新的 `SKILL.md`，再继续执行后续流程；不要继续沿用更新前已经读入的旧说明。若网络不可用、仓库有本地未提交改动，或远端历史无法快进更新，停止并说明原因。

### Step 1. 环境自检

首次使用时运行：

```bash
python3 scripts/check_environment.py
```

脚本优先复用当前环境已有的 `PyMuPDF`。如果缺失，会在首次运行时自动安装到当前 skill 的私有 `.deps/` 目录，不修改用户的全局 Python 环境。首次自动安装需要可访问 Python 包源；如果网络不可用，脚本会给出手动安装命令。

### Step 2. 拆分 PDF 并准备工作目录

```bash
python3 scripts/prepare_pdf_prompts.py input.pdf --out ppt-prompts-output
```

脚本会创建：

- `source_pages/` ——每页一张 PNG，仅供 Agent 分析用。
- `deck_style_brief.md` ——整套 PPT 的统一视觉风格简报（中文，仅供 Agent 内部参考，不作为最终交付物）。
- `visual_optimization_recommendations.md` ——整体视觉优化建议（中文模板，待 Agent 完善）。
- `page_prompts/page-###.md` ——每页的提示词草稿（已标注目标提示词语言，待 Agent 完善）。

**重要说明**：Agent 须视觉识别每页中的图片，并将图片信息融入"版式"字段的布局描述，不单独输出「保留图片」段落。

### Step 3. 完善整体视觉风格简报 `deck_style_brief.md`

Agent 浏览 `source_pages/` 中的每一页，理解整套 PPT 的主题内容与受众定位，然后**为这套 PPT 重新设计一套视觉风格**——原稿的风格和配色一律不得沿用，必须根据内容主题、行业属性、传达目的，从零推导出更合适的新视觉语言。

升级方向涵盖：整体视觉气质（如科技感/人文感/商务感）、全新配色方案（主色/背景色/强调色等少量颜色角色）、字体情绪、空间感与留白原则、装饰元素策略、全套一致性规则。视觉风格可根据内容自由选择，不必刻意追求极简；但装饰元素策略应克制，优先保留能直接支撑主题和信息关系的元素。

在确定统一风格语句前，单独做一次**大方向风格审查**：思考这套内容适合什么气质，然后把这个气质翻译成平面视觉语言。可以有行业定制感、情绪、层次和装饰，但最终写入提示词的风格必须只给大方向（整体气质 + 字体情绪 + 结构母题 + 少量颜色名称与色彩角色），不写纹理、材质、阴影、图标、线条类型、背景图案等细节装饰元素。中文输出，不写十六进制色值。

**关键输出**：在简报末尾单独写出一段「统一风格语句」（1-2 句，包含整体气质 + 字体情绪 + 结构母题 + 2-4 个核心颜色名称与色彩角色，语言与目标提示词语言一致），格式为：

```
统一风格语句：[此处填写，后续每页提示词的风格字段直接复用此句]
```

该语句将被**逐字复制**到每一页提示词的「风格」字段，确保全套风格一致。

### Step 4. 完善整体视觉优化建议 `visual_optimization_recommendations.md`

Agent 必须基于原稿写出**针对本套 PPT 的具体建议**，避免空话。中文输出，覆盖：整体视觉方向、配色系统、字体与信息层级、版式与网格、留白与节奏、图片图标装饰元素、图表与示意图、页面节奏、全局一致性规则、Top 3-5 优先优化动作。

### Step 5. 完善每页的目标语言提示词 `page_prompts/page-###.md`

针对每一页，Agent **必须**用读图（视觉识别）的方式打开 `source_pages/page-###.png`，把页面当作一张图来阅读：

- **直接视觉识别页面文字**：不要假设 PDF 文字层完整可靠。最终内容必须以**视觉识别**的结果为准。
- **确认目标语言**：先视觉浏览全稿，按可见文字粗略判断中文或 English；保持全套单页提示词语言一致，但不要把语言判断结果写入最终提示词正文。
- **识别文字层级**：找出大标题（最主要、字号最大的那一行），以及其下的副标题、小标题、正文段落、数据数字、图注等。
- **识别页面图片**：把页面上所有真实图片/截图/照片/插画/产品图/人像的位置与大致内容记录下来，融入"版式"字段的布局描述中，不单独列为「保留图片」段落。
- **由 Agent 自行决定的元素**：装饰性图标（icon）、几何形状、线条、卡片底纹等——判断可舍弃就直接不写入提示词；判断保留有助于可视化，则在"版式"字段里描述。不要为了让画面显得丰富而堆叠装饰。

**版式思考（每页写提示词前的必经步骤）**：

在动笔写"版式"字段之前，先想清楚：这页内容在讲什么？哪种图形化结构能最直接地传达这个信息？然后做一次视觉减法：一个页面只保留一个核心视觉结构和少量必要辅助元素，删除不承载信息的装饰线、装饰图标、背景纹理、重复几何图形和无意义点阵。接着做一次**幻灯片版式审查**：最终提示词里的版式必须采用简洁的横版幻灯片结构；只给大方向（一个核心视觉结构、主要元素在画面的大致位置、色彩分区），优先使用时间轴、金字塔、流程图、对比分栏、循环图、漏斗图等清晰图形化结构，不写节点细节、图标类型、装饰线、渐变、阴影、纹理等元素级装饰。中文输出，可呼应风格中的主色、背景色和强调色。

**画面简洁度检查**：

- 视觉风格仍由主题、行业和受众决定，不要把“画面简洁”等同于“极简主义风格”。
- 风格描述只给大方向：整体气质 + 字体情绪 + 结构母题 + 2-4 个核心颜色名称与色彩角色；不写十六进制色值，不写纹理、材质、阴影、具体图标、线条类型、背景图案等细节装饰。
- 版式描述只给大方向：一个核心视觉结构 + 主要元素大致位置 + 色彩分区；整体采用简洁扁平的横版幻灯片布局；不写节点细节、图标类型、装饰线、渐变、阴影、纹理等元素级装饰。
- 每页选择一个最能传达语义关系的主结构，例如时间轴、金字塔、流程图、循环图、漏斗图、对比分栏或核心数字 + 注解。
- 辅助元素只用于建立层级、指引阅读路径或解释内容；若删除后不影响理解，就不写入提示词。
- 写完每页最终提示词后复查：如果风格或版式中出现纹理、材质、阴影、具体图标、线条类型、背景图案等细节装饰描述，必须删去，只保留大方向。

**版式字段写作规范**：

"版式"字段用一两句话描述这页要呈现的图形化幻灯片画面——说明采用了什么视觉结构、主要元素在画面的大致位置、色彩分区（可呼应风格中的主色、背景色和强调色）。只给大方向，不写节点细节、图标类型、装饰线、渐变、阴影、纹理等元素级装饰。不写具体参数，留给生图模型自由发挥。最终提示词应使用正向语言描述简洁扁平的幻灯片布局，例如"时间轴结构""金字塔结构""线性流程图""左右分栏""核心数字 + 注解""纯色色块分区"等；不要写"可视化 + 扁平化"这类抽象标签，也不要写纹理或装饰元素。

**风格字段写作规范**：

"风格"字段只给大方向：整体气质 + 字体情绪 + 结构母题 + 2-4 个核心颜色名称与色彩角色（如主色 / 背景色 / 强调色）。中文输出，不写十六进制色值，不把颜色写成冗长色表。不写纹理、材质、阴影、具体图标、线条类型、背景图案等细节装饰。可参考英文「A minimalist, editorial presentation style featuring high-contrast typography (brutalist sans-serif mixed with delicate script), stark black text on an off-white background, and distinctive structural motifs like underscore prefixes and oversized slash-numbering.」的思路，用中文重写为「整体气质 + 字体情绪 + 结构母题 + 少量颜色名称与色彩角色」。

**文字概览写法**：

- **全部文字逐一列出**：把页面上的所有信息文字——标题、副标题、小标题、正文段落、列表项、数据数字、图注、角标等——按层级顺序逐一写出具体内容，不得只写数量描述（如"3 个小标题"）。原稿页码不写入提示词。
- **限制文字生成范围**：文字概览字段必须先写「仅呈现下列明确文字，不添加未列出的文字。」再逐一列出全部文字内容。
  - 正确示范："大标题：「产品交付流程」；副标题：「端到端全链路保障」；步骤标签：需求分析 / 设计评审 / 研发实现 / 上线发布；底部注解：数据来源：XX 报告 2024"
  - 错误示范："大标题 1 行，4 个步骤标签，底部 1 行注解"

**长度与写法（参考 OpenAI GPT-Image 官方提示词指南）**：

- 风格字段只给大方向：整体气质 + 字体情绪 + 结构母题 + 2-4 个核心颜色名称与色彩角色；不写十六进制色值，不写纹理、材质、阴影、具体图标、线条类型、背景图案等细节装饰。中文输出。
- 版式字段只给大方向：一个核心视觉结构 + 主要元素大致位置 + 色彩分区（可呼应风格中的主色、背景色和强调色）；整体采用简洁扁平的横版幻灯片布局；不写节点细节、图标类型、装饰线、渐变、阴影、纹理等元素级装饰。中文输出。
- 用**短标签段**（中文如「风格：…」「版式：…」「文字概览：…」；英文如 `Style:` `Layout:` `Text overview:`），不要写大段散文。
- 顺序遵循 OpenAI 推荐的「场景 → 主体 → 关键细节」：先风格、再版式、再文字概览、最后输出比例。
- 最终提示词中不出现「禁止 xxx」文案——让正向描述覆盖设计意图，不用否定句。

每页提示词必须先用一行「横幅幻灯片」声明图片类型，然后包含 3 段：

1. **风格**：只给大方向（整体气质 + 字体情绪 + 结构母题 + 2-4 个核心颜色名称与色彩角色；不写十六进制色值；不写纹理 / 材质 / 阴影 / 图标 / 线条类型 / 背景图案等细节装饰）。中文输出。
2. **版式**：只给大方向（一个核心视觉结构 + 主要元素大致位置 + 色彩分区；优先时间轴、金字塔、流程图、对比分栏、循环图等简洁图形化幻灯片结构；不写节点细节 / 图标类型 / 装饰线 / 渐变 / 阴影 / 纹理等元素级装饰）。中文输出，可呼应风格中的主色、背景色和强调色。
3. **文字概览 + 输出比例**：先写「仅呈现下列明确文字，不添加未列出的文字。」再把页面上除页码外的全部信息文字逐一列出（按层级顺序写出具体内容）；末尾注明输出比例（横版 16:9）。

### Step 6. 合批输出大提示词

完成所有页面提示词后运行：

```bash
python3 scripts/prepare_pdf_prompts.py --out ppt-prompts-output --finalize
```

脚本会：

- 读取 `page_prompts/page-###.md` 全部页面（按页码顺序）。
- 每 8 页打包成一份 `batched_prompts/batch-01.md`、`batch-02.md`……
- 每份 batch 文件结构精简（**不再在开头重复风格简报和输出比例**——这些信息已经在每页提示词内了）：

  ```
  以下是 N 份提示词，请用可用的生图模型（如 GPT-Image）生成 N 张图片，每份提示词对应一张图片。每张提示词彼此独立，请按顺序逐张生成。

  ## 第 1 张
  （第 1 张的最终提示词正文）

  ## 第 2 张
  （第 2 张的最终提示词正文）

  ……
  ```

- 合批完成后自动删除 `page_prompts/` 目录（该目录为中间产物，不作为最终交付）。

如果用户明确要求只输出提示词、不输出 PDF，或当前不在 Codex 中运行，在这里停止。最终只交付 `batched_prompts/batch-XX.md`；不要生图，不要生成 `redesigned_deck.pdf`。

### Step 7. Codex 默认完整模式：逐页生成图片并合成 PDF

仅当当前平台是 Codex，且用户没有要求只输出提示词或不输出 PDF 时继续本步骤。

先确认 Codex 可以调用内置 `image_gen`。如果不可用，立即停止并告诉用户无法继续生成图片与 PDF。在 Codex 以外的平台不得进入本步骤。

依次读取 `batched_prompts/batch-XX.md` 中每一张的提示词正文。对每一页单独调用一次生图能力，并将返回的图片按顺序保存为：

```
generated_images/page-001.png
generated_images/page-002.png
...
```

生图时使用单页最终提示词原文，不要把多页提示词拼进同一次请求。若某一页失败，只重试该页；若仍无法通过生图能力产出该页，停止生成。禁止改用 PPTX、HTML、SVG、Canvas、网页截图、本地绘图、代码渲染或其他方式补页。全部完成后，检查文件数量和页码连续性，然后运行：

```bash
python3 scripts/assemble_generated_pdf.py --images-dir ppt-prompts-output/generated_images --output ppt-prompts-output/redesigned_deck.pdf
```

### Step 8. 交付给用户

Codex 默认完整模式最终需要交回给用户的产物：

- `visual_optimization_recommendations.md` （整套 PPT 的中文视觉优化建议）
- `redesigned_deck.pdf` （逐页生图后按顺序合成的 PDF，**这是用户主要要的东西**）
- `batched_prompts/batch-XX.md` （每 8 页一份的批量生图大提示词，作为留档）

只输出提示词/不输出 PDF 模式，或 Codex 以外平台使用时，最终只交回：

- `batched_prompts/batch-XX.md` （提示词集合，供用户自行交给生图模型生成图片）

不要返回 PPTX、HTML 渲染、单张图像文件、`page_prompts/` 单页文件、`deck_style_brief.md`、`image_generation_prompts.md` 聚合文档。单张图像是 PDF 的中间产物；`page_prompts/` 在合批后自动清理。

## 何时停止 (Hard Stop Conditions)

- 若用户输入不是 PDF，停止并提示。
- 若 Git clone 安装版无法完成 `scripts/update_skill.py` 更新检查，停止并说明原因。
- 若 PDF 渲染失败，先运行 `python3 scripts/check_environment.py`；若仍失败，停止并把脚本给出的依赖安装命令交给用户。
- 若 Agent 无法看到 `source_pages/` 中的页面截图，必须停止——禁止在缺信息的情况下凭空写提示词。
- 若单页最终提示词语言与全稿目标提示词语言不一致，禁止合批，必须先统一。
- 若提示词里仍残留 `[待补]` `[REPLACE]` `[MISSING]` 等占位符，禁止合批，必须先补齐。
- Codex 默认完整模式下，若无法调用内置 `image_gen`，停止并提示用户无法继续生成图片与 PDF。
- Codex 默认完整模式下，若任一页无法通过 `image_gen` 成功产出，或 `generated_images/` 中页码不连续，禁止合成 PDF，必须先补齐缺失图片。

## 推荐脚本 (Suggested Scripts)

- `scripts/prepare_pdf_prompts.py`
  - 默认模式：拆分 PDF、渲染页面，并生成整体建议、整体风格简报、每页提示词草稿。缺少 `PyMuPDF` 时自动安装私有依赖。
  - `--finalize`：把所有 `page_prompts/page-###.md` 按每 8 页一组合并为 `batched_prompts/batch-XX.md`。
  - `--refresh`：重置已存在的模板（仅在用户希望重写时使用）。
- `scripts/assemble_generated_pdf.py`
  - 检查 `generated_images/page-###.*` 页码连续性，并按顺序合成为 `redesigned_deck.pdf`。优先使用 `PyMuPDF` 或 `Pillow`；均缺失时自动安装私有 `PyMuPDF`。
- `scripts/check_environment.py`
  - 检查当前机器是否具备直接可用的 PDF 拆分和合成后端。
- `scripts/update_skill.py`
  - 下载版安装时跳过更新检查；Git clone 安装时从 GitHub 检查并快进更新当前 skill。
