# PPT 提示词重设计 Lite 版 (中文版，逐页生图并合成 PDF)

本 skill 的目的：为用户提供的 PDF 版 PPT 输出一份**完整的中文视觉优化方案**，包括：

1. 一份整套 PPT 的视觉优化建议文档（中文，针对原稿落地的可执行建议）。
2. 每一页的中文生图提示词，并且只使用 Codex 内置 `image_gen` 能力逐页生成对应图片。
3. 把所有生成图片按页码顺序合成为一个 PDF。
4. 把每一页的提示词按**每 8 页一组**合并成「批量生图大提示词」留档。

**文字处理原则**：页面上所有有信息价值的文字（标题、副标题、正文、列表项、数字、图注等）必须在提示词中逐一列出具体内容，不得只写数量和层级。原稿页码仅用于内部排序，不写入生图提示词，也不出现在最终画面中。

完整的文字内容能让生图模型准确还原信息，同时版式字段负责引导排版创意。

**Lite 版边界**：Lite 只表示版式说明、图片说明和交付文件更轻量，不表示可以省略原稿文字。写提示词时必须优先保证原文完整度；如果某处文字因截图模糊、裁切、乱码而无法可靠识别，必须在当前页提示词前先停止处理并说明需要更清晰来源，不能用"若干文字""几段正文""多个小标题"替代。

## 硬性规则 (Hard Rules)

- 必须使用 Codex 内置 `image_gen` 能力逐页生图：每个 `page_prompts/page-###.md` 的最终提示词单独调用一次，禁止把多页提示词合并为一次生图请求。
- 最终交付 PDF 中的每一页图片必须来自上述 `image_gen` 调用结果；禁止通过生成 PPTX、HTML、SVG、Canvas、网页截图、本地绘图、代码渲染或其他非 `image_gen` 方式制作页面图，再转换成 PDF。
- 若当前 Codex 环境无法调用内置 `image_gen` 能力，必须终止生成并向用户说明原因；不得用其他生成或渲染方式替代。
- 每张生成图片按页码保存到 `generated_images/page-###.png`；必须确认页码连续、数量与原 PDF 页数一致，再合成为 `redesigned_deck.pdf`。
- 拆出的 `source_pages/` 仅供 Agent 分析与填充提示词使用，不会被发送给任何生图工具。
- 最终的视觉优化建议文档必须用中文输出。
- 每一页的生图提示词必须用中文输出。
- **必须通过视觉识别读图**：PDF 文字层提取经常不完整、漏抄或顺序错乱，Agent 必须直接打开 `source_pages/page-###.png` 像看图一样视觉识别整页内容，把页面上的全部文字与全部图片都抄入对应字段——禁止仅依赖 PDF 文字层。
- **图片识别（仅供内部参考，不输出到提示词）**：Agent 须视觉识别页面中的真实图片/截图/照片/插画，并将其位置与内容纳入"版式"字段的布局描述中（如"右侧配合一张产品截图"）——但**不在最终提示词中单独输出「保留图片」段落**。
- **保留范围限定为文字**：原 PPT 中的全部文字必须逐一出现在"文字概览"字段中（不得只写数量/层级，必须写出实际文案内容）。装饰性图标（icon）、几何形状、线条、卡片底纹、渐变背景、分隔条等视觉元素由 Agent 自行判断是否在"版式"字段中描述。
- **每页版式必须图形化**：版式字段必须用视觉结构来传达内容关系，不接受纯文字罗列的排版描述。视觉结构的例子包括但不限于：时间轴、金字塔、循环图、漏斗、同心圆、树状图、维恩图、鱼骨图、雷达图、流程箭头、对比分栏、大数据 + 注解、占比环形图等——应根据内容语义自由选择最合适的结构，不限于此列表。整体保持**扁平化**视觉特征——不要立体、不要 3D 浮块、不要拟物、不要厚重光影。
- **版式必须从内容语义推导，而非套用固定模板**：Agent 在写"版式"字段之前，必须先分析本页的**内容语义**——这页核心信息是什么？信息单元之间是并列、流程、层级、对比、构成还是单一强调关系？——然后从这个内容结构出发，推导出最能直接传达该结构的视觉排版方案。"版式"字段的描述必须体现出「因为内容是 X 关系，所以选择 Y 排版」的因果逻辑，让生图模型感知到排版是内容驱动的，而非随机套用的。
- **排版必须与原图明显不同**：最终生图提示词所描述的排版方案，必须与原 PPT 页面的视觉布局有明显差异——原图中文字的位置、图片的位置、元素的排列方式均**不需要沿用**。Agent 应把原稿中的文字和图片视为「原材料」，而非「参考排版」，完全从内容语义出发重新设计画面结构。如发现写完的版式描述与原图雷同（如同样是左文右图、同样的标题居左、同样的卡片顺序），必须主动调整为更具创意的全新排版方案。
- **提示词只写颜色，不写排版细节参数**：最终生图提示词中，配色（含色值）必须写清楚；但字号大小、字号倍率、百分比占比、具体像素/尺寸、Bold / Regular 等字重描述一律不写，留给生图模型自由发挥。
- **禁止卡片式排版**：最终生图提示词中，不得出现任何卡片式设计的版式描述——包括卡片矩阵、信息卡片、圆角卡片、卡片列表、网格卡片等。对于并列关系的内容，改用无边框的纯色色块分区、分栏对照、图标 + 文字组合、横向并排色带等非卡片方式呈现。
- **全套风格必须一致**：Agent 在 Step 3 完成 `deck_style_brief.md` 后，必须提炼出一段统一的「风格语句」，并将**完全相同**的风格语句复制到每一页提示词的「风格」字段——不得按单页内容随意改变整体视觉调性（如某页改成"科技发布会风"、另一页改成"温暖人文风"）。风格字段的整体定位与气质描述必须全套统一，仅配色的强调色可根据页面内容做轻微调整。
- **提示词中禁止出现任何 PPT 相关描述**：最终生图提示词不得出现「PPT」「幻灯片」「Slide」「演示文稿」「第 X 页」「原 PPT」等字眼，也不得暗示这是某一张幻灯片。每一页提示词必须以**独立海报**的姿态呈现，让生图模型把它当作一张完整的设计作品来生成。
- 不允许在提示词中虚构原稿没有的文字（不发明新标题、副标题、口号、按钮文案、图例等）。
- 必须把所有页的提示词**按每 8 页一组**打包成大提示词文件 `batched_prompts/batch-01.md`、`batch-02.md`……。最后一组不足 8 页时按实际页数生成。
- 单页提示词必须自包含：包含风格（多维度详细描述）、版式（含可视化方案）、文字概览、输出比例。最终提示词中不出现「禁止 xxx」文案。

## 工作流 (Workflow)

### Step 0. 更新 skill 并重新读取说明

每次使用本 skill 时，先运行：

```bash
python3 scripts/update_skill.py
```

如果脚本提示已更新，必须重新读取当前目录下最新的 `SKILL.md`，再继续执行后续流程；不要继续沿用更新前已经读入的旧说明。

自动更新只支持通过 Git clone 安装的 skill。如果用户是通过 ZIP 复制安装，或当前目录没有 Git 远端，脚本会停止并提示用以下方式重新安装：

```bash
git clone https://github.com/Xuebin-Yang/ppt-redesign.git ~/.codex/skills/ppt-redesign
```

如果网络不可用、仓库有本地未提交改动，或远端历史无法快进更新，停止并说明原因。不要在未完成更新检查的情况下继续生成，除非用户明确要求使用当前本地版本。

### Step 1. 环境自检

首次使用时运行：

```bash
python3 scripts/check_environment.py
```

脚本优先复用当前环境已有的 `PyMuPDF`。如果缺失，会在首次运行时自动安装到当前 skill 的私有 `.deps/` 目录，不修改用户的全局 Python 环境。首次自动安装需要可访问 Python 包源；如果网络不可用，脚本会给出手动安装命令。

### Step 2. 拆分 PDF 并准备工作目录

```bash
python3 scripts/prepare_pdf_prompts.py input.pdf --out ppt-prompts-output
```

脚本会创建：

- `source_pages/` ——每页一张 PNG，仅供 Agent 分析用。
- `deck_style_brief.md` ——整套 PPT 的统一视觉风格简报（中文，仅供 Agent 内部参考，不作为最终交付物）。
- `visual_optimization_recommendations.md` ——整体视觉优化建议（中文模板，待 Agent 完善）。
- `page_prompts/page-###.md` ——每页的中文提示词草稿（待 Agent 完善）。

**重要说明**：Agent 须视觉识别每页中的图片，并将图片信息融入"版式"字段的布局描述，不单独输出「保留图片」段落。

### Step 3. 完善整体视觉风格简报 `deck_style_brief.md`

Agent 浏览 `source_pages/` 中的每一页，理解整套 PPT 的主题内容与受众定位，然后**为这套 PPT 重新设计一套视觉风格**——原稿的风格、配色、色值一律不得沿用，必须根据内容主题、行业属性、传达目的，从零推导出更合适的新视觉语言。

升级方向涵盖：整体视觉气质（如科技感/人文感/商务感）、全新配色方案（含主色/辅色/强调色色值）、字体情绪、空间感与留白原则、装饰元素策略、全套一致性规则。视觉风格可根据内容自由选择，不必刻意追求极简；但装饰元素策略应克制，优先保留能直接支撑主题和信息关系的元素。

**关键输出**：在简报末尾单独写出一段「统一风格语句」（1-2 句，包含整体气质 + 全新主配色色值 + 视觉语言），格式为：

```
统一风格语句：[此处填写，后续每页提示词的风格字段直接复用此句]
```

该语句将被**逐字复制**到每一页提示词的「风格」字段，确保全套风格一致。

### Step 4. 完善整体视觉优化建议 `visual_optimization_recommendations.md`

Agent 必须基于原稿写出**针对本套 PPT 的具体建议**，避免空话。中文输出，覆盖：整体视觉方向、配色系统、字体与信息层级、版式与网格、留白与节奏、图片图标装饰元素、图表与示意图、页面节奏、全局一致性规则、Top 3-5 优先优化动作。

### Step 5. 完善每页的中文提示词 `page_prompts/page-###.md`

针对每一页，Agent **必须**用读图（视觉识别）的方式打开 `source_pages/page-###.png`，把页面当作一张图来阅读：

- **直接视觉识别页面文字**：不要假设 PDF 文字层完整可靠。最终内容必须以**视觉识别**的结果为准。
- **识别文字层级**：找出大标题（最主要、字号最大的那一行），以及其下的副标题、小标题、正文段落、数据数字、图注等。
- **识别页面图片**：把页面上所有真实图片/截图/照片/插画/产品图/人像的位置与大致内容记录下来，融入"版式"字段的布局描述中，不单独列为「保留图片」段落。
- **由 Agent 自行决定的元素**：装饰性图标（icon）、几何形状、线条、卡片底纹等——判断可舍弃就直接不写入提示词；判断保留有助于可视化，则在"版式"字段里描述。不要为了让画面显得丰富而堆叠装饰。

**版式思考（每页写提示词前的必经步骤）**：

在动笔写"版式"字段之前，先想清楚：这页内容在讲什么？哪种图形化结构能最直接地传达这个信息？然后做一次视觉减法：一个页面只保留一个核心视觉结构和少量必要辅助元素，删除不承载信息的装饰线、装饰图标、背景纹理、重复几何图形和无意义点阵。把判断结论直接体现在版式描述中，让生图模型感知到排版是为这页内容量身设计的，而非随机套用。布局描述点到为止，留给生图模型自由发挥。

**画面简洁度检查**：

- 视觉风格仍由主题、行业和受众决定，不要把“画面简洁”等同于“极简主义风格”。
- 每页选择一个最能传达语义关系的主结构，例如一条时间轴、一个流程、一组对比分栏或一个核心数字。
- 辅助元素只用于建立层级、指引阅读路径或解释内容；若删除后不影响理解，就不写入提示词。
- 装饰语言用少量点缀表达，不铺满背景，不重复堆叠图标、线条、节点、色块和纹理。

**文字概览写法**：

- **页面主题**：1 句话说明这一页大致讲什么（供 GPT-Image 理解语境，不必出现在画面）。
- **全部文字逐一列出**：把页面上的所有信息文字——标题、副标题、小标题、正文段落、列表项、数据数字、图注、角标等——按层级顺序逐一写出具体内容，不得只写数量描述（如"3 个小标题"）。原稿页码不写入提示词。
  - 正确示范："大标题：「产品交付流程」；副标题：「端到端全链路保障」；步骤标签：需求分析 / 设计评审 / 研发实现 / 上线发布；底部注解：数据来源：XX 报告 2024"
  - 错误示范："大标题 1 行，4 个步骤标签，底部 1 行注解"

**版式字段写作规范**：

"版式"字段用一两句话描述这页要呈现的图形化画面——说明采用了什么视觉结构、主要元素在画面的大致位置。优先描述一个核心视觉结构与必要辅助元素，减少装饰，不把背景填满。整体扁平化，不要立体/3D/拟物/厚重光影。不写具体参数，留给生图模型自由发挥。

**长度与写法（参考 OpenAI GPT-Image 官方提示词指南）**：

- 风格字段需多维度详细描述，涵盖整体视觉气质、配色（含色值）、字体情绪、空间感、装饰语言——写清楚比写短更重要。
- 版式字段须具体描述可视化画面（结构图/流程箭头/分栏对比/时间轴/大字数据/同心圆/占比图……择一），通常只选一个主结构，辅以少量必要元素。
- 用**短标签段**（如「风格：…」「版式：…」「文字概览：…」），不要写大段散文。
- 顺序遵循 OpenAI 推荐的「场景 → 主体 → 关键细节」：先风格、再版式、再文字概览、最后输出比例。
- 最终提示词中不出现「禁止 xxx」文案——让正向描述覆盖设计意图，不用否定句。

每页提示词必须包含 4 段：

1. **生成动作**：`生成一张海报图像（[用本页核心主题关键词描述，如：数据展示/流程说明/核心观点等]）。`
2. **风格**：多维度详细描述（整体气质 + 配色含色值 + 字体情绪 + 空间感 + 装饰语言 + 扁平化）。
3. **版式**：具体描述这一页要呈现的可视化画面，一个核心视觉结构配合少量必要辅助元素，装饰克制，扁平化（不要立体/3D/拟物/厚重光影）。
4. **文字概览 + 输出比例**：页面主题（1 句话）+ 页面上除页码外的全部信息文字逐一列出（按层级顺序写出具体内容）；末尾注明输出比例（横版 16:9）。

### Step 6. 使用 Codex 内置 image_gen 能力逐页生成图片

完成所有页面提示词后，先确认当前环境可以调用 Codex 内置 `image_gen`。如果不可用，立即停止并告诉用户无法继续生成图片与 PDF。

确认可用后，依次读取 `page_prompts/page-###.md` 中 `## 最终提示词` 下方的正文。对每一页单独调用一次 Codex 内置 `image_gen`，并将返回的图片按顺序保存为：

```
generated_images/page-001.png
generated_images/page-002.png
...
```

生图时使用单页最终提示词原文，不要把多页提示词拼进同一次请求。若某一页失败，只重试该页；若仍无法通过 `image_gen` 产出该页，停止生成。禁止改用 PPTX、HTML、SVG、Canvas、网页截图、本地绘图、代码渲染或其他方式补页。全部完成后，检查文件数量和页码连续性，然后运行：

```bash
python3 scripts/assemble_generated_pdf.py --images-dir ppt-prompts-output/generated_images --output ppt-prompts-output/redesigned_deck.pdf
```

### Step 7. 合批输出大提示词

完成所有页面提示词后运行：

```bash
python3 scripts/prepare_pdf_prompts.py --out ppt-prompts-output --finalize
```

脚本会：

- 读取 `page_prompts/page-###.md` 全部页面（按页码顺序）。
- 每 8 页打包成一份 `batched_prompts/batch-01.md`、`batch-02.md`……
- 每份 batch 文件结构精简（**不再在开头重复风格简报和输出比例**——这些信息已经在每页提示词内了）：

  ```
  以下是 N 份提示词，请用 GPT-Image 一次性生成 N 张图片，共 N 张，请按顺序逐张生成。

  ## 第 1 张
  ```

## 何时停止 (Hard Stop Conditions)

- 若用户输入不是 PDF，停止并提示。
- 若 `scripts/update_skill.py` 无法完成更新检查，停止并说明原因；只有用户明确要求使用当前本地版本时才继续。
- 若 PDF 渲染失败，先运行 `python3 scripts/check_environment.py`；若仍失败，停止并把脚本给出的依赖安装命令交给用户。
- 若 Agent 无法看到 `source_pages/` 中的页面截图，必须停止——禁止在缺信息的情况下凭空写提示词。
- 若提示词里仍残留 `[待补]` `[REPLACE]` `[MISSING]` 等占位符，禁止合批，必须先补齐。
- 若当前 Codex 环境无法调用内置 `image_gen`，停止并提示用户无法继续。
- 若任一页无法通过 `image_gen` 生图成功，或 `generated_images/` 中页码不连续，禁止合成 PDF，必须先补齐缺失图片。

## 推荐脚本 (Suggested Scripts)

- `scripts/prepare_pdf_prompts.py`
  - 默认模式：拆分 PDF、渲染页面、生成中文模板（整体建议、整体风格简报、每页提示词草稿）。缺少 `PyMuPDF` 时自动安装私有依赖。
  - `--finalize`：把所有 `page_prompts/page-###.md` 按每 8 页一组合并为 `batched_prompts/batch-XX.md`。
  - `--refresh`：重置已存在的模板（仅在用户希望重写时使用）。
- `scripts/assemble_generated_pdf.py`
  - 检查 `generated_images/page-###.*` 页码连续性，并按顺序合成为 `redesigned_deck.pdf`。优先使用 `PyMuPDF` 或 `Pillow`；均缺失时自动安装私有 `PyMuPDF`。
- `scripts/check_environment.py`
  - 检查当前机器是否具备直接可用的 PDF 拆分和合成后端。
- `scripts/update_skill.py`
  - 在每次使用前从 GitHub 检查并快进更新当前 skill。仅支持 Git clone 安装；ZIP 复制安装无法自动更新。
